# sol-checker — なぜ必要か & 配布・連携デザイン

**作成日**: 2026-03-10
**ステータス**: 承認済み

---

## 1. 課題の構造

```
根本原因：LLM でコードは書けるが、危険性を判断できない非専門家が増えている
    ↓
症状：リスクの存在自体を認識していない
    ↓
結果：資金流出 / 脆弱性放置 / 信頼喪失
```

LLM は Solidity コードを生成できるが、全てのテスト・監査を自動実装するわけではない。
コードを書いたことがない人が LLM でツールを作るケースは今後増え続ける。

## 2. ターゲットユーザー

### Primary: LLM でコードを書く非専門家
- Web3 に興味を持った非エンジニア（ビジネス側、デザイナー、マーケター等）
- ChatGPT に「クラファンのコントラクト作って」と頼んでデプロイまでしてしまう人
- 「監査」という概念自体を知らない、または LLM にコストを割けない層

### Primary (自分自身)
- sol-checker の一番のユーザーは作者自身
- crypthub.app をはじめ、今後作る Web3 サービス全ての安全網
- 自分の開発で得た知見をクイズコンテンツに還元するループ

## 3. sol-checker の位置づけ

**「あなたのコードにはこういう危険がある」を教える最初の警告**

- 教育的アプローチ：なぜ危険か（impact）+ どう直すか（fix）
- 将来有料候補：「LLM にこのプロンプトを投げれば修正できます」というガイド付きレポート
  - 混乱すると人は「どんなプロンプトで聞けば解決するか」がわからなくなる
  - プロンプト自体が商品価値になる

## 4. 料金モデル

- **当面は完全無料** — ユーザーが自分の API キー（ANTHROPIC_API_KEY）を設定して使う
- コストはユーザー側に発生、作者側にはかからない
- **有料化トリガー**: ホスティング型（API キーを sol-checker 側で提供）に移行し、API コストが自分にかかるようになった時のみ

## 5. crypthub.app 連携

### コンセプト
物語クイズの中で主人公に失敗を体験させ、自然に sol-checker へ誘導する。

### シナリオ: クラファンコントラクトの失敗
- 主人公が LLM でクラウドファンディングコントラクトを作成
- デプロイ後、reentrancy 攻撃で資金を抜かれる
- クラファンは誰でもイメージしやすく、「お金を集める → 盗まれる」のインパクトが直感的

### 二段構えクイズ
1. **デプロイ前**: コードを見せて「このコントラクト、デプロイして大丈夫？」
   - 多くの人が「大丈夫」と答えてしまう
2. **事故後**: 「主人公の資金が盗まれました。原因は何？」
   - 選択肢から脆弱性を当てさせる
   - 解説で「なぜ危険か」を教育

### 導線 2 ルート
1. **宣伝サイト → クイズ**: 「こんな事故が起きますよ」と予告 → クイズへ誘導
2. **クイズ → sol-checker**: 失敗体験 → 解説 → 「防ぐツールあります。お試しあれ！」
